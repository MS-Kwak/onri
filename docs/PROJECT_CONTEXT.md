## 1. 서비스 개요

**한 줄 정의:** 결이 맞는 사람끼리 안전하게 연결되는 성소수자 친화 매칭 앱

**서비스명:** 온리 (영문: `Onri`)
**슬로건:** "온전한 나로 쉬어가는 곳"
**의미:** 순우리말 '온(온전한, 전부의)'에서 출발한 조어. 차별과 편견에 지친 마음이
있는 그대로의 자기 자신으로 안식하고 자연스럽게 사람과 이어지는 공간. 'only'와의
연상으로 "당신을 위한 유일한 곳"이라는 뉘앙스도 포함.

**핵심 컨셉:**

- 매칭·데이팅이 코어. 커뮤니티·정보교류는 연결된 사용자 사이에서 자연 발생하는 부가가치.
- 기존 앱(예: 특정 성향으로 쏠린 서비스)의 검증된 수익 구조(소액 채팅 개시 모델)는
  계승하되, "채팅 시도 남발"이 아니라 "결이 맞는 건강한 연결"에 유인을 둔다.
- **안전·신뢰(특히 아웃팅 방지)가 핵심 경쟁력.**

**타겟 전략:**

- 플랫폼 구조는 성소수자 전체 수용 (FTM/MTF/논바이너리/시스 등).
- 단, 초기 커뮤니티 시딩은 **FTM·트랜스 중심**으로 집중 (색깔이 특정 성향으로
  굳는 것을 방지).

### 타이포 / 톤

- 담백하고 중성적이며 모던한 톤. 특정 성별·성향에 치우치지 않는다.
- 따뜻함(치유·안식)과 세련됨(안전·신뢰)이 공존.
- CTA는 골드 배경 + 네이비 텍스트가 기본 조합.

### 반복 컴포넌트 (화면정의서에서 도출된 공통 컴포넌트)

- **Button** (Primary: 골드/네이비, Secondary: 크림, Outline: 골드 보더)
- **Pill / Chip** (필터 칩, 정체성 태그, 상태 뱃지)
- **ProfileCard** (썸네일 · 닉네임 · 요약 · 하트 버튼)
- **BottomTab** (홈 · 수락함 · 채팅 · MY)
- **ChatBubble** (좌: 상대 크림/보더, 우: 본인 골드)
- **Card** (구성 섹션용 흰 카드 + 얇은 보더 + 약한 그림자)

## 3. 핵심 정책 (반드시 준수)

| 항목      | 정책                                   | 이유                                       |
| --------- | -------------------------------------- | ------------------------------------------ |
| 가입 연령 | **만 19세 이상** (성인 인증 필수)      | 청소년 보호, 앱스토어 심사, 운영자 책임    |
| 인증 방식 | **휴대폰 본인인증** (PASS/통신사)      | 확실한 성인·본인 확인, 중복·악의 가입 억제 |
| 수익 모델 | **하트 충전제** (소액, MVP는 이것만)   | 검증된 결제 구조, MVP 단순화               |
| 무료 하트 | **출석체크로 지급** + 연속 출석 보너스 | 리텐션 유도 + 대량 발송 억제               |
| 재화명    | **하트** (heart)                       | 직관적, 설명 불필요                        |

### 안전·법적 요구사항 (개발 시 항상 고려)

- **아웃팅 방지가 최우선.** 성적 지향·성정체성은 **개인정보보호법상 민감정보** →
  별도의 명시적 동의 + 암호화 저장 + 최소 수집.
- 프로필 각 항목의 **노출 범위를 사용자가 직접 제어**할 수 있어야 함.
- 신고·차단·강제탈퇴 기능은 관련 화면(프로필/채팅)에 **항상 접근 가능**하게.
- Supabase **RLS(Row Level Security)로 민감정보 접근 제어** — "누가 어떤 필드를
  보는가"를 DB 레벨에서 통제.

## 4. 기술 스택 / 아키텍처

| 영역           | 기술                                          | 비고                                   |
| -------------- | --------------------------------------------- | -------------------------------------- |
| 웹/전체 개발   | **Next.js 15** (App Router)                   | 전 기능을 Next.js로 개발               |
| 스타일         | **Tailwind CSS**                              | 브랜드 색상을 config에 토큰으로 등록   |
| 폰트           | **Pretendard**                                | 담백·중성적·모던 톤에 부합하는 한글 폰트 |
| 아이콘         | **Lucide React**                              | 가볍고 일관된 라인 아이콘              |
| 상태 관리      | **Zustand**                                   | 클라이언트 전역 상태 (하트 잔액, 인증 등) |
| 백엔드·DB      | **Supabase** (Auth · DB · Realtime · Storage) | 인증, 데이터, 실시간 채팅, 저장        |
| 모바일 앱      | **React Native (WebView 래핑)**               | Next.js를 WebView로 감싸 배포          |
| 네이티브       | **푸시 알림(FCM) · 딥링크만 네이티브 브릿지** | 재방문 핵심 동력. 나머지는 WebView     |
| 언어           | **TypeScript**                                |                                        |
| 패키지 매니저  | **npm**                                       |                                        |

### 아키텍처 원칙

- **Next.js로 전 기능 개발 → React Native로 WebView 래핑.**
  하트·수락·메시지 도착 등 재방문 유도 알림(FCM)과 딥링크만 네이티브로 구현.
- **실시간 채팅은 Supabase Realtime** — WebView 환경에서도 동작하므로 별도
  네이티브 작업 불필요.
- **개발 전략: Mock-data-first.** Supabase 연동 전에 목업 JSON으로 UI를 먼저
  완성하고, 이후 데이터 소스만 교체한다.
- **모바일 퍼스트 반응형.** 375px 기준으로 설계, WebView 래핑 전제.
- **다크/라이트 테마 모두 지원.** 다크(Navy 배경) 기본, 라이트(Paper 배경) 선택 가능.
  시스템 설정에 따라 자동 전환 + 수동 토글.

## 5. 화면 구성 (MVP 8개 화면)

핵심 흐름: **온보딩 → 본인인증 → 정체성 설정 → 홈 탐색 → 프로필 상세 → 하트 보내기 → 수락 → 채팅**

각 화면의 라우트는 App Router 기준 예시이며, 실제 구현 시 조정 가능.

### 01. 스플래시 / 온보딩 `/` 또는 `/onboarding`

- 구성: 로고·슬로건, 소셜 로그인(카카오/Apple), 이메일 로그인
- 동작: 자동 로그인(세션 유효 시 홈), 신규→본인인증 / 기존→홈 분기
- 예외: 인증 취소 시 스플래시 유지, 네트워크 오류 토스트

### 02. 본인인증 (만 19세) `/auth/verify`

- 구성: 인증 폼(통신사·이름·주민번호 앞자리·휴대폰), 인증번호(SMS+타이머), 약관 동의
- 동작: 성인 검증(만 19세 미만 즉시 차단), 본인확인기관(PASS/통신사) 연동
- 예외: 미성년자 가입 불가, 인증 실패 재시도, 중복 가입 시 로그인 유도

### 03. 정체성 / 지향 설정 `/onboarding/profile`

- 구성: 정체성 선택(FTM/MTF/논바이너리/트랜스/시스 등), 관계 목적(연애·친구·정보교류, 복수),
  노출 범위 제어
- 동작: 단계형 프로그레스, 최소 수집(필수만, 건너뛰기 허용)
- 예외/정책: 성적 지향은 **별도 동의·암호화 저장**, 가입 후 언제든 수정 가능

### 04. 홈 (탐색 + 필터) `/home`

- 구성: 필터 바(정체성·관계목적·지역·연령대 칩), 프로필 카드(썸네일·닉네임·요약·하트),
  하단 탭(홈·수락함·채팅·MY)
- 동작: 무한 스크롤, 카드에서 빠른 하트 전송, 인증·활동 프로필 노출 우선
- 예외: 결과 없음(필터 완화 제안 empty state), 하트 부족(충전/출석 유도)

### 05. 프로필 상세 `/profile/[id]`

- 구성: 프로필 정보(사진·닉네임·나이·인증뱃지·소개), 정체성 태그(공개 설정된 것만),
  하단 고정 '하트 보내기' CTA
- 동작: 하트 전송(잔여 차감→상대 알림), 신고/차단(우상단 메뉴 즉시 접근)
- 예외: 이미 전송('수락 대기중'), 차단 관계(접근 불가), 하트 부족(모달)

### 06. 매칭 / 수락함 `/matches`

- 구성: 탭(받은 하트/보낸 하트), 요청 카드(썸네일·요약·수락/거절), 상태(대기중·수락됨·만료)
- 동작: 수락 시 즉시 채팅방 생성 후 채팅 탭 이동, 거절 시 목록 제거·상대엔 미노출
- 예외: 요청 없음(홈 탐색 유도), 무응답 자동 만료

### 07. 1:1 채팅 `/chat/[roomId]`

- 구성: 대화 영역(말풍선·시간·읽음), 상단 바(상대 정보·신고/차단), 안전 배너(상시)
- 동작: Supabase Realtime 실시간 전송, 메시지 도착 시 FCM 푸시
- 예외/안전: 차단 시 대화 중단, 신고 접수(대화 첨부), 스크린샷 경고

### 08. 마이페이지 `/my`

- 구성: 하트 잔액·충전 진입, 출석체크(7일 그리드·연속 보너스), 설정 메뉴(알림·차단목록·약관·로그아웃)
- 동작: 출석 지급(1일 1회+연속 추가), 소액 인앱결제 충전(5천/1만원 등)
- 예외: 중복 출석(버튼 비활성), 결제 실패(재시도)

## 6. 데이터 모델 힌트 (목업 → 추후 Supabase 테이블 초안)

UI를 목업으로 만들 때 아래 구조를 참고. 이것이 향후 DB 테이블 설계의 출발점이 된다.
**민감 필드는 RLS로 접근 제어할 대상임을 표시(⚠️).**

```ts
// User / Profile
type Profile = {
  id: string;
  nickname: string;
  age: number;
  region: string;
  thumbnailUrl: string;
  isVerified: boolean; // 본인인증 완료 여부
  identity: Identity; // ⚠️ 민감정보
  lookingFor: RelationGoal[]; // 연애 | 친구 | 정보교류
  bio: string;
  visibility: {
    // 항목별 노출 범위 (사용자 제어)
    identity: 'public' | 'private';
    region: 'public' | 'private';
  };
  createdAt: string;
};

type Identity =
  | 'FTM'
  | 'MTF'
  | 'NONBINARY'
  | 'TRANS'
  | 'CIS'
  | 'OTHER'; // ⚠️
type RelationGoal = 'DATING' | 'FRIEND' | 'INFO';

// Heart (재화 + 전송 기록)
type HeartBalance = { userId: string; balance: number };
type HeartSend = {
  id: string;
  fromUserId: string;
  toUserId: string;
  status: 'PENDING' | 'ACCEPTED' | 'DECLINED' | 'EXPIRED';
  createdAt: string;
};

// Match / Chat
type ChatRoom = {
  id: string;
  userIds: [string, string];
  createdAt: string;
};
type Message = {
  id: string;
  roomId: string;
  senderId: string;
  text: string;
  readAt: string | null;
  createdAt: string;
};

// 출석
type Attendance = {
  userId: string;
  date: string; // YYYY-MM-DD
  streak: number; // 연속 출석일
  rewardedHearts: number;
};
```

## 7. 현재 진행 상태 & 다음 작업

**완료:**

- 서비스 기획서 (브랜드·정책·수익모델·기술스택 확정)
- 브랜드 네이밍: 온리(Onri) + 로고(네이비+골드 SVG)
- 화면정의서 (8개 화면 + 와이어프레임 + 예외처리)

**다음 (이 문서를 받은 시점의 작업):**

1. **프로젝트 셋업** — Next.js 15(App Router) + TypeScript + Tailwind + Pretendard +
   Lucide React + Zustand. 브랜드 색상을 `tailwind.config`에 토큰으로 등록.
   다크/라이트 테마 모두 설정.
2. **디자인 시스템 / 공통 컴포넌트** — Button, Pill, ProfileCard, BottomTab,
   ChatBubble, Card 먼저 구축.
3. **목업 데이터로 8개 화면 UI 구현** (mock-data-first).
4. **라우팅 연결** — 위 화면 라우트대로 App Router 구성.
5. (이후) **Supabase 연동** — Auth, DB, Realtime + RLS 정책.

**작업 원칙:**

- 항상 브랜드 팔레트(2절)와 정책(3절)을 준수.
- Supabase 붙이기 전 목업으로 UI 완성.
- 민감정보(⚠️) 취급 코드는 노출 범위·접근 제어를 항상 고려.
- 화면정의서의 예외처리(empty state, 에러, 부족 상태)를 UI에 반영.
- 모바일 퍼스트(375px 기준)로 설계.
